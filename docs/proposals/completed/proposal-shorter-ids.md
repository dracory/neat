## Current State of Primary Keys

Neat ORM currently generates primary keys exclusively through **database auto-increment mechanisms**. The system is hardcoded around integer-based IDs (`uint`, `int`, and their sized variants).

### How Primary Keys Are Generated Today

#### 1. Schema Builder Layer
The `Blueprint.ID()` method creates an auto-incrementing unsigned big integer:

```go
func (r *Blueprint) ID(column ...string) schema.ColumnDefinition {
    if len(column) > 0 {
        return r.BigIncrements(column[0])
    }
    return r.BigIncrements("id")
}
```

This translates per database driver as:

| Database | SQL Type |
|----------|----------|
| **SQLite** | `INTEGER PRIMARY KEY AUTOINCREMENT` |
| **MySQL** | `BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY` |
| **PostgreSQL** | `BIGSERIAL PRIMARY KEY` (or `GENERATED ALWAYS AS IDENTITY`) |
| **SQL Server** | `BIGINT IDENTITY(1,1) PRIMARY KEY` |
| **Oracle** | `NUMBER(20) GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY` |

#### 2. Model Layer
The base `orm.Model` struct defines the ID as `uint`:

```go
type Model struct {
    ID uint `json:"id"`
    Timestamps
}
```

The ORM identifies primary keys by reflection, looking strictly for fields named `ID` or `Id`.

#### 3. Insert / ID Retrieval Layer
After `Create()`, the ORM retrieves the generated ID and populates the model:

- **PostgreSQL**: appends `RETURNING id` to the INSERT, scans the returned `int64`.
- **SQL Server**: uses `OUTPUT INSERTED.id`, scans the returned `int64`.
- **Oracle**: attempts sequence lookups (`TABLENAME_SEQ`, `SEQ_TABLENAME`, `TABLENAME_ID_SEQ`), then falls back to `SELECT MAX(id) FROM table`.
- **MySQL / SQLite**: uses `sql.Result.LastInsertId()`.

The helper `setModelPrimaryKey(value any, id int64)` then writes the ID back into the struct field. All primary key helpers (`getPrimaryKeyValue`, `setModelPrimaryKey`, `getModelPrimaryKey`) are locked to **integer types only**.

#### 4. Query Builder Layer
`BuildInsert` hardcodes logic around integer IDs. The SQL Server `OUTPUT` clause specifically looks for columns named `id`, `user_id`, `post_id`, or `uuid` -- but the retrieved value is still treated as `int64`.

#### 5. Save Logic
`ToSql.Save()` decides insert vs update by calling `getPrimaryKeyValue(value)`: if the result is non-zero, it updates; otherwise it inserts. This is a pure integer-zero check.

## Proposal: Add Support for Short String-Based IDs

### Goal
Allow developers to opt into **short, URL-friendly, string-based primary keys** (e.g., `1jkne4e0jv0` -- 11 lowercase characters) instead of database auto-increment integers, while preserving full backward compatibility with the existing integer-based default.

### Design Principles

1. **Backward Compatible**: Existing models using `orm.Model` with `ID uint` continue to work exactly as before.
2. **Opt-in**: Short IDs are enabled per-model via a new base struct or tag convention.
3. **Database Agnostic**: Short IDs are generated in Go code before insert, so no database-specific auto-increment configuration is needed.
4. **Thread-Safe**: ID generation must be safe under concurrent use.

---

### Proposed API

#### Option A: Composable ShortID Trait (Recommended)

Introduce a minimal embeddable struct that **only** provides the short ID field. Users compose their own models explicitly:

```go
package orm

// ShortID provides a short string primary key field.
// Embed it in your model to opt into client-generated short IDs.
// No timestamps, no soft deletes — just the ID.
type ShortID struct {
    ID string `json:"id" db:"id"`
}
```

Usage — compose exactly what you need:

```go
// With timestamps
type Product struct {
    orm.ShortID
    orm.Timestamps
    Name  string  `db:"name"`
    Price float64 `db:"price"`
}

// Without timestamps
type Category struct {
    orm.ShortID
    Name string `db:"name"`
}

// With soft deletes too
type Order struct {
    orm.ShortID
    orm.Timestamps
    orm.SoftDeletes
    Total float64 `db:"total"`
}
```

The ORM detects `ShortID` by reflection (field named `ID` of kind `string`) and generates an ID before insert when the field is empty.

#### Option B: Tag-Based Opt-In

For users who prefer not to embed anything, a struct tag can mark the field:

```go
type Product struct {
    ID    string `json:"id" db:"id" neat:"shortId"`
    Name  string `db:"name"`
}
```

The tag acts as an explicit signal; the ORM treats any `string` `ID`/`Id` field with `neat:"shortId"` as a short-ID column.

### ID Generation Implementation

Instead of an external dependency, we vendor a small internal utility package inside Neat. The implementation is ~60 lines.

Create `support/uid/uid.go`:

```go
package uid

import (
    "strings"
    "sync"
    "time"
)

// Crockford Base32 alphabet (omits I, L, O, U to avoid ambiguity).
const crockfordAlphabet = "0123456789abcdefghjkmnpqrstvwxyz"

var (
    idMutex sync.Mutex
)

// GenerateShortID creates a new 11-character lowercase short ID.
// It encodes the current microsecond timestamp in Crockford Base32.
// Thread-safe via mutex to prevent duplicate IDs under concurrency.
func GenerateShortID() string {
    idMutex.Lock()
    defer idMutex.Unlock()

    ts := time.Now().UnixMicro()
    return strings.ToLower(encodeCrockford(ts))
}

// encodeCrockford encodes a positive int64 into Crockford Base32.
func encodeCrockford(n int64) string {
    if n == 0 {
        return "0"
    }
    var buf [13]byte // max length for int64 in base32
    i := len(buf) - 1
    for n > 0 {
        buf[i] = crockfordAlphabet[n&0x1f]
        n >>= 5
        i--
    }
    return string(buf[i+1:])
}

// NormalizeID normalizes an ID to lowercase for consistent lookups.
func NormalizeID(id string) string {
    return strings.ToLower(strings.TrimSpace(id))
}

// IsShortID checks whether the given string matches short ID lengths (11 or 21 chars).
func IsShortID(id string) bool {
    length := len(id)
    return length == 11 || length == 21
}
```

> **Why internal?** The logic is trivial (timestamp + base32 encode) and keeping it inside Neat removes an external dependency, improves build reproducibility, and lets us tweak the alphabet or length without waiting for upstream releases.

### Required ORM Changes

#### 1. Primary Key Helpers

Update `database/query/utils.go` to handle **string** primary keys:

```go
// getPrimaryKeyValue -- existing signature remains for ints,
// add a new helper that returns (any, bool) to support both int and string.
func getPrimaryKeyValueAny(value any) (any, bool) {
    v := reflect.ValueOf(value)
    if v.Kind() == reflect.Ptr {
        if v.IsNil() {
            return nil, false
        }
        v = v.Elem()
    }
    if v.Kind() != reflect.Struct {
        return nil, false
    }
    for _, name := range []string{"ID", "Id"} {
        field := v.FieldByName(name)
        if !field.IsValid() {
            continue
        }
        switch field.Kind() {
        case reflect.Uint, reflect.Uint8, reflect.Uint16, reflect.Uint32, reflect.Uint64:
            u := field.Uint()
            if u > 0 {
                return int64(u), true
            }
            return int64(u), true
        case reflect.Int, reflect.Int8, reflect.Int16, reflect.Int32, reflect.Int64:
            return field.Int(), true
        case reflect.String:
            return field.String(), true
        }
    }
    return nil, false
}

// isPrimaryKeyZero reports whether the primary key is unset.
func isPrimaryKeyZero(value any) bool {
    v := reflect.ValueOf(value)
    if v.Kind() == reflect.Ptr {
        if v.IsNil() {
            return true
        }
        v = v.Elem()
    }
    if v.Kind() != reflect.Struct {
        return true
    }
    for _, name := range []string{"ID", "Id"} {
        field := v.FieldByName(name)
        if !field.IsValid() {
            continue
        }
        switch field.Kind() {
        case reflect.Uint, reflect.Uint8, reflect.Uint16, reflect.Uint32, reflect.Uint64:
            return field.Uint() == 0
        case reflect.Int, reflect.Int8, reflect.Int16, reflect.Int32, reflect.Int64:
            return field.Int() == 0
        case reflect.String:
            return field.String() == ""
        }
    }
    return true
}
```

#### 2. Create Method

In `database/query/query_create.go`, before building the INSERT, check if the model has an empty string `ID`. If so, generate a short ID and set it:

```go
func (q *Query) Create(value any) error {
    // ... existing validation and Creating event ...

    // If the model uses a string primary key and it's empty, generate one.
    if isShortIDModel(value) {
        // Detection: the model has an `ID` field of kind `string`
        // (e.g. from embedded `orm.ShortID`) or has `neat:"shortId"` tag.
        if id, ok := getPrimaryKeyValueAny(value); ok {
            if s, _ := id.(string); s == "" {
                setModelPrimaryKey(value, GenerateShortID()) // new string-aware setter
            }
        }
    }

    // ... existing BuildInsert / exec logic ...

    // Skip LastInsertId / RETURNING logic for short-ID models
    // because the ID is already known client-side.
}
```

> **Important**: When a short string ID is pre-generated, the ORM must **not** overwrite it with `LastInsertId` or `RETURNING` results, and the `BuildInsert` logic must include the `id` column in the INSERT rather than omitting it.

#### 3. Save Method

Update `ToSql.Save()` to use `isPrimaryKeyZero()` instead of `getPrimaryKeyValue(value) != 0`:

```go
func (t *ToSql) Save(value any) string {
    if isPrimaryKeyZero(value) {
        // INSERT
        builder := NewBuilder(t.query)
        sql, args := builder.BuildInsert(value)
        // ...
    } else {
        // UPDATE
        // ...
    }
}
```

#### 4. Builder Insert

Ensure `BuildInsert` includes the `id` column when it has a non-empty string value (currently it may skip zero-value fields). The `extractColumnsAndValues` logic already skips zero values; this needs to be adjusted so that an explicitly set short ID is treated as a value to insert.

#### 5. Schema Builder

Add a new Blueprint helper for short-ID columns:

```go
func (r *Blueprint) ShortID(column ...string) schema.ColumnDefinition {
    name := "id"
    if len(column) > 0 {
        name = column[0]
    }
    return r.String(name, 21).Primary() // 21 chars max for Crockford-shortened IDs
}
```

Migration usage:

```go
db.Schema().Create("products", func(table neat.Blueprint) {
    table.ShortID()            // VARCHAR(21) PRIMARY KEY
    table.String("name", 255)
    table.Timestamps()
})
```

### Migration Path for Existing Projects

1. **New tables**: Compose `orm.ShortID` + `table.ShortID()` from day one.
2. **Existing tables**: No changes required. Integer auto-increment remains the default.
3. **Converting existing tables**: Would require a manual migration to change the column type and backfill IDs. This is considered out of scope for the initial proposal.

### Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| **Collision** | `uid.TimestampMicro` is monotonic; mutex prevents duplicate generation within the same process. Across distributed processes, timestamp granularity (microseconds) plus Crockford encoding makes collisions practically impossible. |
| **Breaking integer PK logic** | Keep `getPrimaryKeyValue` for backward compat; add new `any`-returning helpers only where needed. All existing tests must pass unchanged. |
| **Database portability** | String PKs work on all supported databases (SQLite, MySQL, PostgreSQL, SQL Server, Oracle). No driver-specific code needed for short IDs. |
| **Sortability** | Short IDs encode a timestamp, so they are roughly sortable by creation time (not perfectly, but close enough for most use cases). If strict sortability is needed, a separate `created_at` index should be used. |

### Acceptance Criteria

- [ ] `orm.ShortID` can be embedded in a struct and produces a non-empty 11-character ID on `Create()`.
- [ ] Existing models using `orm.Model` (`ID uint`) continue to auto-increment exactly as before.
- [ ] `Save()` correctly performs an INSERT when the string `ID` is empty, and an UPDATE when it is set.
- [ ] Bulk insert works for short-ID models (each element gets a unique generated ID).
- [ ] All existing unit and integration tests pass without modification.

### Alternatives Considered

1. **UUID (github.com/google/uuid)** -- Already in `go.mod`, but UUIDs are 36 characters long and not URL-friendly.
2. **NanoID / CUID** -- Popular in JavaScript ecosystems, but add external dependencies and are not timestamp-based.
3. **KSUID / ULID** -- Excellent sortable IDs, but typically 27 characters; Crockford-shortened TimestampMicro gives us 11 characters with similar properties.
4. **Snowflake IDs (int64)** -- Twitter-style distributed integers; still numeric and require a coordination service.

The Crockford-shortened timestamp approach was chosen because it is already used in sibling projects (`coursestore`), is extremely short (11 chars), is sortable, and requires no central coordination. By vendoring the logic inside `support/uid/` we keep the ORM self-contained.
