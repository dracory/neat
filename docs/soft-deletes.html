Soft Deletes

Soft deletes mark records as deleted without removing them from the database. The row remains in the table and can be restored. Normal queries automatically exclude soft-deleted records.

What are Soft Deletes?

Soft deletes are useful for:

  • Data recovery and audit trails
  • Compliance requirements (data retention)
  • Undo / restore workflows

Soft Delete Strategies

Neat supports two soft delete strategies to accommodate different database requirements and performance needs:

1. NULL-Based Strategy (Default)

The traditional approach using NULL to indicate active records:

  • Active when: soft_deleted_at IS NULL
  • Deleted when: soft_deleted_at IS NOT NULL
  • Column constraint: Nullable
  • Default value: NULL

This strategy is simple, widely-used, and easy to understand. It's the default choice for most applications.

2. Max-Date Sentinel Strategy

An alternative approach using a far-future timestamp as a sentinel value:

  • Active when: soft_deleted_at > NOW() (future sentinel value)
  • Deleted when: soft_deleted_at <= NOW() (past or present)
  • Column constraint: NOT NULL (recommended)
  • Default value: 9999-12-31 23:59:59 UTC

This strategy offers better compatibility with NOT NULL constraints and improved index performance through range scans instead of NULL checks.

Strategy Comparison

Aspect NULL-Based Max-Date Sentinel
"Not deleted" value NULL 9999-12-31 23:59:59
Column constraint Nullable NOT NULL (recommended)
Default value NULL 9999-12-31 23:59:59
Active query condition IS NULL > NOW()
Deleted query condition IS NOT NULL <= NOW()
Index efficiency Varies by database Better (range scans)
NOT NULL compatible No Yes
Simplicity Very simple Slightly more complex

Choosing a Strategy

Use NULL-Based when:

  • You need simplicity and clarity
  • NOT NULL constraints are not required
  • You want the most common and widely-understood pattern
  • Database index performance is not a critical concern

Use Max-Date Sentinel when:

  • You need NOT NULL constraints on the soft delete column
  • You need better index performance on large datasets
  • Working with databases that have poor NULL index handling
  • You want to avoid NULL handling complexity in queries

Choosing an Embed

Neat provides built-in soft delete embeds for both strategies. Pick the one that matches your schema:

NULL-Based Embeds

Embed Column When to use
soft_delete.SoftDeletes soft_deleted_at Default — new projects
soft_delete.SoftDeletedAt soft_deleted_at Explicit naming (mirrors CreatedAt/UpdatedAt)
soft_delete.DeletedAt deleted_at Laravel-compatible or existing deleted_at schemas

Max-Date Sentinel Embeds

Embed Column When to use
soft_delete.SoftDeletesMaxDate soft_deleted_at Max-date strategy with NOT NULL support
soft_delete.DeletedAtMaxDate deleted_at Laravel-compatible with max-date strategy

All embeds are in the github.com/dracory/neat/database/soft_delete package. Equivalent sql.NullTime variants exist in github.com/dracory/neat/database/orm.

Setting Up

1. Add the embed to your model

NULL-Based Strategy:

import "github.com/dracory/neat/database/soft_delete"

// New project — default column soft_deleted_at
type User struct {
    soft_delete.SoftDeletes
    ID   uint
    Name string
}

// Laravel-compatible — column deleted_at
type Post struct {
    soft_delete.DeletedAt
    ID    uint
    Title string
}

Max-Date Sentinel Strategy:

import "github.com/dracory/neat/database/soft_delete"

// Max-date strategy with soft_deleted_at column
type Product struct {
    soft_delete.SoftDeletesMaxDate
    ID    uint
    Name  string
    Price float64
}

// Laravel-compatible with max-date strategy
type Article struct {
    soft_delete.DeletedAtMaxDate
    ID    uint
    Title string
}

2. Add the column to your schema

For NULL-Based Strategy:

-- SoftDeletes / SoftDeletedAt
ALTER TABLE users ADD COLUMN soft_deleted_at DATETIME NULL;

-- DeletedAt
ALTER TABLE posts ADD COLUMN deleted_at DATETIME NULL;

Or using the schema builder:

table.Timestamp("soft_deleted_at").Nullable()
table.Timestamp("deleted_at").Nullable()

For Max-Date Sentinel Strategy:

-- SoftDeletesMaxDate / DeletedAtMaxDate
ALTER TABLE users ADD COLUMN soft_deleted_at DATETIME NOT NULL DEFAULT '9999-12-31 23:59:59';

ALTER TABLE posts ADD COLUMN deleted_at DATETIME NOT NULL DEFAULT '9999-12-31 23:59:59';

Or using the schema builder:

table.Timestamp("soft_deleted_at").Default("9999-12-31 23:59:59")
table.Timestamp("deleted_at").Default("9999-12-31 23:59:59")

Note: The max-date sentinel value is available as a constant:

import "github.com/dracory/neat/database/soft_delete"

// MaxSoftDeletedAt = 9999-12-31 23:59:59 UTC
fmt.Println(soft_delete.MaxSoftDeletedAt)

Soft Deleting Records

// Via query builder — sets soft_deleted_at = NOW() in the database
db.Query().Model(&User{}).Where("id = ?", 1).SoftDelete()

// Via struct method — sets the field in memory (does not hit the database)
user.SoftDelete()

Querying

var users []User

// Default — excludes soft-deleted records (soft_deleted_at IS NULL)
db.Query().Model(&User{}).Get(&users)

// Include all records (including soft-deleted)
db.Query().Model(&User{}).WithSoftDeleted().Get(&users)

// Only soft-deleted records
db.Query().Model(&User{}).OnlySoftDeleted().Get(&users)

// Explicitly exclude (reset a previous WithSoftDeleted)
db.Query().Model(&User{}).WithoutSoftDeleted().Get(&users)

Restoring Records

// Restore soft-deleted records
db.Query().Model(&User{}).Where("id = ?", 1).Restore()

// Restore via struct method
user.Restore()

Force Delete (Permanent Deletion)

// Force delete (permanently remove from database)
db.Query().Model(&User{}).Where("id = ?", 1).ForceDelete()

// Force delete via struct method
user.ForceDelete()

Checking if Soft Deleted

// Check if record is soft deleted
if user.IsSoftDeleted() {
    // Record is soft deleted
}
