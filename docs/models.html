Models and Struct Tags

This document explains how Neat ORM maps Go structs to database tables and columns.

Important: Two Separate Concepts

Neat has two separate systems for working with database tables:

  1. Struct Tags - Map existing database columns to Go struct fields
  2. Schema Builder - Define and create database table structures

Struct tags do NOT define database types. They only tell Neat which database column maps to which struct field.

To create tables with specific column types (VARCHAR, TEXT, INT, etc.), use the Schema Builder in migrations.

Table Names

Neat determines table names in the following order:

  1. TableName() method - If your model implements TableName() string, that method is used
  2. Automatic naming - The struct name is converted to snake_case and pluralized

Custom Table Name

type User struct {
    ID   uint
    Name string
}

func (User) TableName() string {
    return "app_users"
}

Automatic Table Name

type User struct {
    ID   uint
    Name string
}
// Maps to table: "users"

type UserProfile struct {
    ID     uint
    UserID uint
    Bio    string
}
// Maps to table: "user_profiles"

Column Names

Neat determines column names by checking struct tags in this priority order:

  1. db tag - used directly as column name
  2. neat tag - used directly as column name
  3. gorm tag - parses column:name format for GORM compatibility
  4. Automatic naming - field name converted to snake_case

Using db Tags (Recommended)

type User struct {
    ID        uint      `db:"id"`
    Name      string    `db:"name"`
    Email     string    `db:"email"`
    CreatedAt time.Time `db:"created_at"`
    UpdatedAt time.Time `db:"updated_at"`
}

Using neat Tags

type User struct {
    ID        uint      `neat:"id"`
    Name      string    `neat:"name"`
    Email     string    `neat:"email"`
}

Using gorm Tags (For Compatibility)

type User struct {
    ID        uint      `gorm:"primaryKey;column:id"`
    Name      string    `gorm:"size:255;column:name"`
    Email     string    `gorm:"uniqueIndex;column:email"`
}

Automatic Column Names (No Tags)

type User struct {
    ID        uint      // maps to "id"
    Name      string    // maps to "name"
    Email     string    // maps to "email"
    CreatedAt time.Time // maps to "created_at"
    UpdatedAt time.Time // maps to "updated_at"
    IsActive  bool      // maps to "is_active"
}

How They Work Together

Step 1: Create the Table (Schema Builder)

First, use a migration to define the table structure with explicit database types:

// migration file
db.Schema().Create("users", func(table neat.Blueprint) {
    table.ID()                    // BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY
    table.String("name", 255)     // VARCHAR(255)
    table.String("email", 500)    // VARCHAR(500)
    table.Text("bio")             // TEXT
    table.Integer("age")          // INT
    table.Boolean("active")       // BOOLEAN
    table.DateTime("created_at")  // DATETIME
    table.DateTime("updated_at")  // DATETIME
})

Step 2: Define the Struct (Struct Tags)

Then, create a Go struct that maps to those columns:

type User struct {
    ID        uint      `db:"id"`         // maps to "id" column (BIGINT)
    Name      string    `db:"name"`       // maps to "name" column (VARCHAR)
    Email     string    `db:"email"`      // maps to "email" column (VARCHAR)
    Bio       string    `db:"bio"`        // maps to "bio" column (TEXT)
    Age       int       `db:"age"`        // maps to "age" column (INT)
    Active    bool      `db:"active"`     // maps to "active" column (BOOLEAN)
    CreatedAt time.Time `db:"created_at"` // maps to "created_at" column (DATETIME)
    UpdatedAt time.Time `db:"updated_at"` // maps to "updated_at" column (DATETIME)
}

Key Points

  • Go types (uint, string, int, bool, time.Time) are for Go code only
  • Database types (BIGINT, VARCHAR, TEXT, INT, BOOLEAN, DATETIME) are defined in Schema Builder
  • Struct tags (db:"name") just connect the two - they don't define types
  • Neat handles type conversion between Go and database types automatically

Primary Keys

Neat automatically looks for fields named ID or Id as the primary key.

Integer Primary Keys (Default)

By default, Neat expects uint or int primary keys that are auto-incremented by the database.

type User struct {
    ID    uint // Primary key
    Name  string
    Email string
}

Short String Primary Keys (Opt-In)

For URL-friendly, client-generated primary keys, embed orm.ShortID instead of orm.Model. The ID is generated automatically before insert using a Crockford Base32-encoded microsecond timestamp (11 lowercase characters, e.g., 1jkne4e0jv0).

import "github.com/dracory/neat/database/orm"

type Product struct {
    orm.ShortID      // ID string — generated client-side
    orm.Timestamps   // optional — add only if you need it
    Name  string     `db:"name"`
    Price float64    `db:"price"`
}

Use table.ShortID() in migrations for string-primary-key tables:

db.Schema().Create("products", func(table neat.Blueprint) {
    table.ShortID()            // VARCHAR(21) PRIMARY KEY
    table.String("name", 255)
    table.Decimal("price", 10, 2)
    table.Timestamps()
})

Model Embeds

Neat provides several embed structs to add common functionality to your models:

Base Model

import "github.com/dracory/neat/database/orm"

type User struct {
    orm.Model  // Includes ID and timestamps
    Name  string `db:"name"`
    Email string `db:"email"`
}

Timestamps

import "github.com/dracory/neat/database/orm"

type User struct {
    ID    uint             `db:"id"`
    Name  string           `db:"name"`
    orm.Timestamps       // created_at and updated_at
}

Individual Timestamps

import "github.com/dracory/neat/database/orm"

type User struct {
    ID        uint       `db:"id"`
    Name      string     `db:"name"`
    orm.CreatedAt       // Only created_at
    orm.UpdatedAt       // Only updated_at
}

Soft Deletes

import "github.com/dracory/neat/database/soft_delete"

type User struct {
    soft_delete.SoftDeletes  // soft_deleted_at column
    ID    uint               `db:"id"`
    Name  string             `db:"name"`
}

For Laravel compatibility:

import "github.com/dracory/neat/database/soft_delete"

type User struct {
    soft_delete.DeletedAt  // deleted_at column
    ID    uint             `db:"id"`
    Name  string           `db:"name"`
}

Model Relationships

Define relationships using association methods in your models:

type User struct {
    ID    uint   `db:"id"`
    Name  string `db:"name"`
    Posts []Post `db:"-"` // Don't map to database column
}

type Post struct {
    ID      uint   `db:"id"`
    Title   string `db:"title"`
    UserID uint   `db:"user_id"`
    User    User   `db:"-"`
}

Model Scopes

Scopes allow you to define reusable query constraints:

type User struct {
    ID     uint   `db:"id"`
    Name   string `db:"name"`
    Active bool   `db:"active"`
    Premium bool  `db:"premium"`
}

// Define scopes
func ActiveUsers(q Query) Query {
    return q.Where("active", true)
}

func PremiumUsers(q Query) Query {
    return q.Where("premium", true)
}

// Use scopes
var users []User
db.Query().Model(&User{}).Scopes(ActiveUsers, PremiumUsers).Get(&users)

Model Observers

Observers allow you to hook into model lifecycle events:

type UserObserver struct{}

func (o *UserObserver) Creating(event neat.Event) error {
    user := event.GetModel().(*User)
    // Validate before creation
    return nil
}

func (o *UserObserver) Created(event neat.Event) error {
    user := event.GetModel().(*User)
    // Send welcome email
    return nil
}

// Register observer
db.Orm().Observe(&User{}, &UserObserver{})

Model Serialization

Neat automatically handles JSON serialization based on struct tags:

type User struct {
    ID        uint   `json:"id" db:"id"`
    Name      string `json:"name" db:"name"`
    Email     string `json:"email" db:"email"`
    Password  string `json:"-" db:"password"` // Hidden from JSON
    CreatedAt time.Time `json:"created_at" db:"created_at"`
}

// JSON output
userJSON, _ := json.Marshal(user)
// {"id":1,"name":"John","email":"john@example.com","created_at":"2024-01-01T00:00:00Z"}

Model Validation

Use observers to implement validation logic:

func (o *UserObserver) Creating(event neat.Event) error {
    user := event.GetModel().(*User)
    
    if user.Name == "" {
        return errors.New("name is required")
    }
    
    if !isValidEmail(user.Email) {
        return errors.New("invalid email format")
    }
    
    return nil
}

Model Type Conversion

Neat automatically handles type conversion between Go and database types:

Go Type Database Type
uint, int INT, BIGINT, etc.
string VARCHAR, TEXT, CHAR
bool BOOLEAN, TINYINT
time.Time DATETIME, TIMESTAMP, DATE
float64 FLOAT, DOUBLE, DECIMAL
[]byte BLOB, BINARY

Model Query Builder Integration

Use models with the query builder for type-safe operations:

var user User
err := db.Query().Model(&User{}).Where("id", 1).First(&user)

var users []User
err := db.Query().Model(&User{}).Where("active", true).Get(&users)

err := db.Query().Model(&User{}).Create(&user)

err := db.Query().Model(&User{}).Where("id", 1).Update(&user)

result, err := db.Query().Model(&User{}).Where("id", 1).Delete()

Model Factory Integration

Use models with the factory for test data generation:

factory := db.Factory()
factory.Model(&User{}).Count(10).Create(&User{})

Advanced Model Features

Hidden Attributes

Use struct tags to hide sensitive fields from serialization:

type User struct {
    ID       uint   `json:"id" db:"id"`
    Name     string `json:"name" db:"name"`
    Password string `json:"-" db:"password"` // Hidden from JSON
}

Custom Attribute Names

Use different names for Go fields vs database columns:

type User struct {
    ID       uint   `db:"id"`
    FullName string `db:"full_name" json:"full_name"`
    Email    string `db:"email" json:"email_address"`
}

Computed Fields

Define methods for computed properties:

type User struct {
    FirstName string `db:"first_name"`
    LastName  string `db:"last_name"`
}

func (u *User) FullName() string {
    return u.FirstName + " " + u.LastName
}

Model Best Practices

  • Use db tags for explicit column mapping
  • Use json tags for API serialization control
  • Embed common functionality (Timestamps, SoftDeletes, etc.)
  • Use observers for validation and lifecycle hooks
  • Use scopes for reusable query logic
  • Keep models simple - business logic in services
  • Use associations for relationships instead of manual joins
